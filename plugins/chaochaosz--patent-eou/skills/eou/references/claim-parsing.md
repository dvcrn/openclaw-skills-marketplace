## 1. Identifying Independent vs. Dependent Claims

- **Independent claim**: Does NOT reference any other claim. Example: "A device comprising..."
- **Dependent claim**: References another claim. Example: "The device of claim 1, wherein..."

For EOU analysis, focus on **independent claims** first — they define the full scope of protection.
Dependent claims can be analyzed if the user wants broader coverage mapping.

## 2. Claim Decomposition Rules

Split a claim into elements at these linguistic boundaries:
- Each "comprising" / "including" / "having" introduces a new element list
- Semicolons (`;`) separate parallel elements
- "wherein" clauses are sub-elements — attach to their parent element
- "whereby" clauses describe results, not limitations — lower weight for infringement

### Example Decomposition

**Raw claim text:**
> "A wireless communication device comprising: a processor configured to encode data using a first encoding scheme; a transmitter coupled to the processor, wherein the transmitter is configured to transmit the encoded data over a licensed frequency band; and a memory storing instructions that, when executed by the processor, cause the processor to switch to a second encoding scheme based on channel conditions."

**Decomposed:**
```
[1a] A wireless communication device comprising:
[1b] a processor configured to encode data using a first encoding scheme;
[1c] a transmitter coupled to the processor,
     [1c-i] wherein the transmitter is configured to transmit the encoded data
            over a licensed frequency band;
[1d] a memory storing instructions that, when executed by the processor,
     cause the processor to switch to a second encoding scheme
     based on channel conditions.
```

## 3. Claim Types and What to Look For

| Claim Type | What it protects | Key evidence target |
|---|---|---|
| **Apparatus/Device** | A physical product or system | Hardware specs, block diagrams, component lists |
| **Method/Process** | Steps of a process | User manuals, flow diagrams, API docs, operational descriptions |
| **System** | Multiple components working together | System architecture, integration docs |
| **CRM (Computer-readable medium)** | Software/firmware stored on a medium | Firmware descriptions, software architecture |

## 4. Technical Term Disambiguation

When a claim uses technical terms:
1. Check the patent **specification (description section)** for the inventor's own definition
2. If defined in the spec → use that definition (it controls over ordinary meaning)
3. If not defined → use the ordinary meaning in the relevant technical field
4. Check the **prosecution history** (if available) for any claim amendments that narrow scope

## 5. Claim Scope Signals

**Broad scope indicators** (favor finding infringement):
- "comprising" (open-ended — product can have additional features)
- "at least one"
- "one or more"
- "configured to" (functional language — broad)

**Narrow scope indicators** (harder to find infringement):
- "consisting of" (closed — product cannot have additional features)
- "exactly N"
- Specific numeric values or materials
- "wherein said [element] is [very specific description]"

## 6. Method Claim Special Rule

For method claims, infringement requires that **all steps are performed**.
- If performed by a single entity → direct infringement
- If steps split across two parties (e.g., vendor + user) → induced or contributory infringement (note this in the report)

## 1. Evidence Source Hierarchy

Use sources in this priority order for credibility:

| Priority | Source Type | Why |
|---|---|---|
| 1 | Official product datasheets / technical specifications | Authoritative, company-published |
| 2 | User manuals / operation guides | Describes actual functionality |
| 3 | Patent filings by the accused company | Admissions about their own technology |
| 4 | FCC / regulatory filings | Legally certified technical disclosures |
| 5 | Standards body submissions (ETSI, IEEE, 3GPP) | Public technical disclosures |
| 6 | Developer documentation / API docs | Implementation details |
| 7 | Press releases / marketing materials | Use for supplementary support only |
| 8 | Third-party teardowns / reverse engineering reports | Use with appropriate caveats |

## 2. Mapping Each Claim Element

For each claim element, follow this process:

### Step A: Extract the functional essence
Strip the legal language and ask: **"What does this element actually do or require?"**

Example:
- Claim element: "a processor configured to encode data using a first encoding scheme"
- Functional essence: "Some component that performs encoding"

### Step B: Search for the function in product materials
Use these search strategies:
- **Keyword search**: Use both the claim's technical terms AND synonyms/product-specific terms
- **Synonym expansion**: encoder → codec, coder, compression module, DSP
- **Block diagram lookup**: Often reveals component architecture matching claim structure
- **Index/glossary search**: Check if product docs have a glossary with relevant terms

### Step C: Collect and record verbatim evidence
- Quote the product document directly (short excerpt)
- Record: source name, page number or URL, section heading
- If from web: record full URL and access date

### Step D: Assess match type

#### DIRECT match criteria:
The product evidence uses the same (or clearly synonymous) terminology and describes the same functionality as the claim element.

#### EQUIVALENT match criteria (Doctrine of Equivalents):
Ask the **Function-Way-Result (FWR) test**:
- Does the product perform substantially the **same function**?
- In substantially the **same way**?
- Achieving substantially the **same result**?
If yes to all three → EQUIVALENT

#### NOT MET criteria:
- No evidence of the feature in any available product documentation
- Product explicitly uses a different approach that does not satisfy FWR test
- Claim requires a specific component/step that product provably lacks

#### UNCLEAR criteria:
- Feature may exist but is not publicly documented
- Marketing material hints at capability but no technical detail available
- Architecture diagrams show the system but implementation details are absent

## 3. Multi-Source Search Strategy

### 3A. Search Source Hierarchy

When conducting searches, prioritize sources in this order for credibility and depth:

| Tier | Source | How to Access |
|---|---|---|
| **Tier 1 — Technical Authoritative** | Official datasheets, spec sheets, technical whitepapers | `web_search "[product] datasheet"` + `web_fetch` |
| **Tier 1 — Regulatory** | FCC Equipment Authorization Database | `web_search "[product] FCC ID"` or `web_fetch fcc.gov/oet/ea` |
| **Tier 1 — Patent Admissions** | Accused company's own patent filings | `web_search site:patents.google.com [company] [tech term]` |
| **Tier 2 — Standards** | ETSI IPR, IEEE, 3GPP technical contributions | `web_search "[tech term] ETSI 3GPP IEEE submission"` |
| **Tier 2 — Developer Docs** | SDK docs, API references, developer guides | `web_search "[product] developer documentation [tech term]"` |
| **Tier 3 — Market Intelligence** | TechInsights, IHS Markit, Gartner reports | `web_search "TechInsights [product] teardown [tech term]"` |
| **Tier 3 — Teardowns** | iFixit, AnandTech, Chipworks analysis | `web_search "iFixit [product] teardown"` |
| **Tier 4 — Press & Marketing** | Press releases, product pages, launch videos | Use only as supplementary corroboration |

### 3B. Specialized Database Access

When APIs are configured, query these databases directly:
- **USPTO / Google Patents**: Full patent text, cited prior art, continuation families
- **Lens.org**: Open patent database, citation analysis, legal status
- **FCC OET**: Equipment authorization, RF test reports, hardware block diagrams
- **ETSI IPR Portal**: Standards-essential patent declarations
- **PatSnap / Derwent Innovation**: Patent landscape, assignee portfolios
- **IFI CLAIMS**: Statistical patent data
- **TechInsights / iSuppli**: Professional teardown and chip analysis reports

When these APIs are **not available**, use `web_search` to find:
- Public excerpts and summaries from these sources
- News articles citing their reports
- Conference papers and journal articles referencing the same data

### 3C. Web Search Execution for Product Evidence

When conducting searches per product, run this structured sequence:

```
Core product evidence searches:
1. "[Product Name] datasheet"
2. "[Product Name] technical specification"
3. "[Product Name] [key claim term]" (repeat for each major claim element)
4. "[Company Name] [Product Name] whitepaper"
5. "[Product Name] FCC ID" (for wireless/hardware devices)
6. "[Company Name] patent [technology area]" (for admission evidence)
7. "[Product Name] how [feature] works" (for implementation details)
8. "[Company Name] [technology term] blog engineering" (for engineering blogs)
9. "site:patents.google.com [company] [technology keyword]" (for company patent admissions)
10. "[product] teardown iFixit chip [technology component]" (for hardware products)
```

Always use `web_fetch` to retrieve full page content — search snippets are often insufficient for claim element mapping.

## 4. Handling Ambiguous Evidence

### If evidence is marketing language only:
> "Product X features advanced AI-powered encoding technology"

→ Mark as UNCLEAR, note: "Marketing claim suggests encoding capability but no technical implementation details available."

### If evidence is in a foreign language:
→ Translate, then use. Note: "[Translated from Japanese datasheet]"

### If evidence is from a teardown/third-party:
→ Use with caveat: "Per third-party analysis [source], the product contains [component], which suggests..."

Standard citation format for the report:

For web pages:
> [Source: [URL], accessed [date], §"Technical Specifications"]

For user-provided text:
> [Source: User-provided product description]

## 6. Common Pitfalls to Avoid

| Pitfall | Correct approach |
|---|---|
| Accepting marketing language as technical evidence | Note as supplementary; seek datasheet confirmation |
| Ignoring negative evidence (product explicitly does NOT do X) | Record as NOT MET with the counter-evidence |
| Treating "could" as "does" | Only map features that product actually implements, not potential capabilities |
| Over-relying on equivalents | EQUIVALENT should be argued with FWR test; don't use it to paper over weak evidence |
| Missing method steps | For method claims, verify ALL steps are performed, not just most |
